# Standby: Web Search UI Improvements & Privacy (2026-04-15)

This document captures findings and concerns regarding the **Web Search UI** to be addressed after the current codebase migration/shrinking phase.

## 1. Web Search Result Persistence (Bug)

### Issue
Web search results (links and activity log) disappear from the UI the moment the AI starts responding.

### Findings
- **Trigger**: In `Dashboard.tsx`, `isThinking` is set to `false` when the first token of the AI response arrives.
- **Rendering**: In `ChatArea.tsx`, the `SearchStatus` component is conditionally rendered only when `isThinking` is `true`.
- **State**: `researchEvents` in `Dashboard.tsx` is transient and cleared on each new message.

### Proposed Solution
- **Metadata Integration**: Attach the `researchEvents` array to the `metadata` object of the specific AI `Message` in the `messages` store.
- **Persistent Rendering**: Move `SearchStatus` inside the `MessageBubble` or render it as a sibling to the message text based on the message's own metadata, ensuring it remains visible even after the response is complete.

## 2. Favicon Privacy Concerns

### Issue
The current implementation fetches favicons from an external service (`icons.duckduckgo.com`), which leaks the domains the AI is visiting to a third party.

### Concerns
- Using **Google S2** or other external services is a privacy compromise for a "privacy-forward" application like Finch.
- Total lack of icons makes the UI feel unpolished.

### Proposed Solution: "Privacy-First Fallback"
- **Zero-Leakage Icons**: If a favicon is missing or to avoid external pings entirely, generate a stylized fallback icon locally.
- **Implementation**:
  - Create a colored circle using a deterministic hash of the domain name (HSL color).
  - Display the first letter (initial) of the domain in the center.
  - This ensures a colorful, premium-feeling UI without sending any domain metadata to third-party servers.

## 3. Multimodal & File Upload System (Phase 2)

### Issue
The current "File Tool" is a UI-only skeleton.

### Proposed Architecture (Provider-Agnostic)

#### 1. Internal Representation (Rust IR)
To handle discrepancies between providers, the Rust core will use a unified message structure:
```rust
enum ContentPart {
    Text(String),
    Image { base64: String, mime: String },
    Document { base64: String, mime: String }, // Primarily for PDFs
}
struct InternalMessage {
    role: String,
    content: Vec<ContentPart>,
}
```

#### 2. Provider Adapter Mapping
The backend will translate the `InternalMessage` using the following logic:

| Provider | Block Type | Field Name | Data Format |
| :--- | :--- | :--- | :--- |
| **Anthropic** | `image` / `document` | `source` | `{"type": "base64", "data": "..."}` |
| **OpenAI** | `image_url` | `image_url` | `{"url": "data:image/png;base64,..."}` |
| **Gemini** | `inlineData` | `inlineData` | `{"mimeType": "...", "data": "..."}` |
| **Ollama** | `image` (array) | `images` | List of raw base64 strings |

#### 3. Data Pipeline
- **Small IPC**: Frontend sends `[{ type: "file", path: "/path/to/img.png" }]`.
- **Just-In-Time Encoding**: Rust reads from disk and encodes to base64 *only* when the outbound HTTP request is being built for that specific provider's adapter.

## 4. Artifacts System (Phase 3)

### Concept
Finch will intercept specific XML-like tags in the AI's response to render content in a dedicated "Artifact Sidebar."

### Protocol Definition
We will follow the Claude convention by listening for the `<antArtifact>` tag:
```xml
<antArtifact 
  identifier="unique-id" 
  title="Artifact Title" 
  type="application/vnd.ant.code" 
  language="typescript"
>
  [Content]
</antArtifact>
```

### Frontend Implementation
- **Regex Interceptor**: A hook that scans the incoming stream for `<antArtifact>` markers.
- **Side Panel**: When an artifact is detected, a sliding side-panel opens to render the code (using Shiki), Markdown, or a preview.
- **Persistence**: Artifacts should be stored in the message metadata so they can be reopened from the chat history.

## 5. Persona Selection & Multi-User Architecture

### Concept
Finch will move away from a single "Jane Doe" profile to a Persona-based system. On launch, the user is presented with a Netflix-style selection screen to choose their active "Mindset."

### Technical Architecture
- **Persona Context**: Each Persona is an object containing:
    - `name` and `email`
    - `avatar_path` (local file)
    - `system_prompt` (The persona's specialized personality/instructions)
    - `default_model` & `provider`
    - `ui_preferences` (Theme, backgrounds, etc.)
- **Workflow**:
    - **Launch**: Mandatory Persona selection (with "Auto-login to this persona" setting).
    - **Quick Swap**: Hotkey (`Cmd+Shift+P`) to open a switcher and swap personas instantly.
    - **Isolation**: Each persona has its own chat history (optional/configurable).

## 6. Hardware & State Orchestration (Warm Standby)

### Refinement of Inactivity Timer
The current "Full Eject" on inactivity is too aggressive. We will implement a "Hardware-Only Unload":
- **UI State Persistence**: The `selectedModel` and `selectedProvider` remain active in the UI after the 10-minute timer.
- **VRAM Purge**: Only the `eject_model` command is sent to the Rust backend (Ollama/LM Studio) to free system resources.
- **The "Red Border"**: The UI uses the existing `isModelLoaded` polling to show a red border around the input area, signaling the model is "Sleeping."
- **Automatic Wake-Up**: Typing or sending a message automatically re-triggers the model load.

## 7. Profile Isolation & Data Protection

### Storage Separation
- **`finch_config.json`**: Sensitive API keys and provider endpoints.
- **`user_profile.json`**: User identity, personas, and preferences.

### Data Protection (Anti-Corruption)
- **Atomic Writes**: All JSON config updates will follow the `Write Temp -> Verify -> Rename` pattern to prevent half-written/corrupted files.
- **Shadow Backups**: Maintain a `.bak` copy of the last known-good state. If the primary file is malformed, Finch will automatically recover from the backup.

## 8. Progressive Identity & The "Semantic Vault" (In-Progress)

### Goal
Implement a high-security, high-usability "Contact Intelligence" system that functions like a Password Manager (1Password/Bitwarden) but for your entire professional and personal identity.

### The "Vault-Style" Architecture
Instead of a flat JSON, Personas will be modeled as a structured vault of attributes with per-field metadata: `value`, `confidence`, `source`, `last_verified`, `scope` (Work/Home), and `sensitivity`.

### The 5-Step "Review & Commit" Flow
To ensure data safety and zero-leakage, every "Autofill" or "Update" follows this pipeline:
1. **Intent Detection**: The AI detects a task (e.g., "Fill out this job app").
2. **Inference**: The model identifies which persona attributes are likely required (Name, GitHub, Bio).
3. **Vault Matching**: Finch matches these requirements against the stored Persona attributes.
4. **The Three-State Review UI**:
    - **`Known & Matched`**: High confidence. Ready to fill.
    - **`Ambiguous`**: Multiple candidates found (e.g., Work vs Personal Email). User must pick from a ranked list.
    - **`Missing`**: Data doesn't exist. AI offers to generate it or prompts the user.
5. **Explicit Commit**: Data is only "injected" into the prompt/form after a physical user click.

### Conflict & Overwrite Logic (The "Diff View")
When suggesting an update to existing data (e.g., a new Phone Number), Finch will present an explicit **Comparison Modal**:
- Shows **[Current Value]** vs **[Proposed Value]**.
- Provides per-field checkboxes to allow partial updates.
- Displays a confidence score/rationale for why the change was suggested.

### Performance & Privacy Guardrail
- **On-Demand Retrieval**: Profile data is retrieved via a tool only when relevant.
- **Privacy Firewall**: Sensitive fields (PII) are redacted by default for Frontier models unless the "Commit" action is triggered.
